# Data Model: Atomic Team Reorder

No new tables. No new columns. No new persisted fields. This document restates the existing entities, the relationships the recalculate reads, and the per-scope invariant the recalculate maintains.

## Entities (existing)

### `Team` (`public.Teams` — `libs/backend/database/src/models/team.model.ts`)

Relevant fields for this feature:

| Field          | Type        | Notes                                                                                                                                                  |
| -------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `id`           | UUID PK     | Stable. Tie-breaker when two teams share `baseIndex` (R3).                                                                                             |
| `clubId`       | UUID FK     | One leg of the scope key.                                                                                                                              |
| `season`       | int         | One leg of the scope key.                                                                                                                              |
| `type`         | enum        | `M` / `F` / `MX` / `NATIONAL`. Drives tier membership; pooling rule (R1) decides whether a recalculate touches one or two tiers.                       |
| `teamNumber`   | int         | Written ONLY by the recalculate. Default `1`. Initial value from `createTeam` is a placeholder, overwritten on next recalculate.                       |
| `name`         | varchar     | Auto-derived. Regenerated by the model's `BeforeUpdate` hook whenever `teamNumber` or `type` changes (`team.model.ts:232–238`).                        |
| `abbreviation` | varchar     | Same as `name` — auto-derived, regenerated by the same hook.                                                                                           |
| `link`         | varchar     | Cross-season continuity key. Untouched by this feature.                                                                                                |

Relevant relations:

- `team.memberships` (`HasMany TeamPlayerMembership`, `team.players` reaches them via the through-association). The base-member subset feeds `baseIndex` (R2).
- `team.club` — only used by `Team.generateName` to format the display string.

### `TeamPlayerMembership` (`public.TeamPlayerMemberships`)

Relevant fields: `teamId`, `playerId`, `membershipType`, `start`, `end`. The recalculate considers only memberships in the team's base set when computing `baseIndex`.

### `Player` (`public.Players`) and `RankingLastPlace` (`public.RankingLastPlaces`)

The recalculate loads `RankingLastPlace` rows for the players in each team's base set, scoped to the primary `RankingSystem`, to populate the `IndexPlayer` shape that `getIndexFromPlayers` consumes (`single`, `double`, `mix`, `gender`). Same path as `enrollment.service.ts:210–211`.

### `RankingSystem` (`ranking.RankingSystems`)

`findOne({ where: { primary: true } })` returns the system whose `amountOfLevels` is passed into `getIndexFromPlayers`. One row per primary system; existing data.

## Derived value (transient, not persisted)

### `baseIndex(team) -> number`

```text
basePlayers   = team.memberships filtered to base-set memberships
indexPlayers  = basePlayers joined to RankingLastPlace (primary system) -> { single, double, mix, gender }
baseIndex     = getIndexFromPlayers(team.type, indexPlayers, system.amountOfLevels)
```

Lower is stronger (R2). Computed on every recalculate call. Not stored. Not exposed on the GraphQL `Team` type by this feature.

## Recalculate scope (the unit of one mutation call)

Driven by the resolver from `(type, nationalCountsAsMixed)`:

| `type` | `nationalCountsAsMixed` | Resolves to scope `types` (in tier order) | Lock key |
| --- | --- | --- | --- |
| `M` | (any) | `[M]` | `M` |
| `F` | (any) | `[F]` | `F` |
| `MX` | `false` | `[MX]` | `MX` |
| `MX` | `true` | `[NATIONAL, MX]` | `MX+NAT` |
| `NATIONAL` | (any) | `[NATIONAL]` | `MX+NAT` |

The lock key is canonicalized so any call that touches NATIONAL teams in the same `(clubId, season)` collides (R4).

## Invariant (enforced)

For every recalculate call with scope `types = [T1, T2, …]` against `(clubId, season)`:

1. Let `tierTeams[i] = sort(teams of type T_i, by=(baseIndex ASC, id ASC))`.
2. After commit, `tierTeams[0][j].teamNumber == j + 1` for `j ∈ [0, |tierTeams[0]|)`.
3. After commit, `tierTeams[i][j].teamNumber == (Σ_{k<i} |tierTeams[k]|) + j + 1` for every subsequent tier `i ≥ 1` and every `j`.
4. Every tier's teams have `name` and `abbreviation` regenerated to match the new number; neither contains `_temp`.

For non-recalculate code paths (`updateTeam`, `createTeam`, `createTeams`, `deleteTeam`, ranking sync), `teamNumber` / `name` / `abbreviation` MUST NOT be written. (The `createTeam` placeholder block is the only place outside the recalculate where `teamNumber` may be assigned, and the recalculate overwrites it on its next run.)

## What this feature does NOT change

- The `BeforeUpdate` model hook (`team.model.ts:232–245`). Continues to regenerate `name`/`abbreviation` whenever `teamNumber` changes. The recalculate's `team.save` call relies on this hook.
- The `Team.generateName` / `Team.generateAbbreviation` helpers.
- The `(link, season)` idempotency key on `createTeam` (002-team-resolver-improvements).
- The `EventEntry` / `TeamPlayerMembership` lifecycle.
- The `TeamResult` GraphQL object type (`team-result.object.ts`).
- The `nationalCountsAsMixed` flag and pooling logic in `createTeam`'s placeholder MAX+1 block.

## Migration footprint

- **No schema migration**. The advisory-lock approach requires no new tables, columns, or indexes.
- **No data migration**. The legacy `_temp` artifact only exists in local/staging and self-heals on the next recalculate against the affected scope (R6).
