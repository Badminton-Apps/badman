# Implementation Plan: Stabilise SearchPlayer under broad search terms

**Branch**: `027-fix-search-player-n1` | **Date**: 2026-05-19 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/027-fix-search-player-n1/spec.md`

## Summary

A two-line GraphQL request from the new frontend (`SearchPlayer` with a 2-letter iLike) crashes the API because (a) the `players` resolver has no server-side row cap and (b) the `Player.rankingLastPlaces` field resolver fires one SQL per parent row. Fix is two narrow edits, both following patterns already merged on `main`:

1. Cap the `players` query at default `take = 25` / max `take = 200` server-side.
2. Introduce `PlayerAssociationService` (request-scoped, `dataloader@2.x`) that batches `Player → RankingLastPlace[]` for the primary system, and rewrite `rankingLastPlaces` to delegate to it.

No schema change. No new dependency. No frontend coordination. Mirrors `TeamAssociationService` (merged via PR #923 / spec 019).
