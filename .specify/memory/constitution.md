# Badman Constitution

## Core Principles

### I. Code-First GraphQL via Sequelize Models

Domain entities are defined exactly once, as Sequelize model classes that double
as Apollo `@ObjectType` declarations. New persistent entities MUST be authored
this way; separate hand-written GraphQL SDL or duplicate DTO classes are
prohibited. Input types MUST be derived via `PartialType` / `OmitType` from the
same model class. Models MUST be placed under the correct schema namespace
(`public`, `event`, `ranking`, `security`, `system`, `personal`).

**Rationale**: A single source of truth for entities eliminates drift between
ORM and API layers and is the architectural backbone of `@badman/backend-graphql`.

### II. Translation Discipline (NON-NEGOTIABLE)

All changes to `packages/backend-translate/assets/i18n/{en,nl_BE,fr_BE}/all.json` —
including additions, edits, renames, moves, flattening, and merges — MUST be
performed via the `translation-manager` agent. Hand edits, shell one-offs
(`jq`, `node -e`, `sed`, bulk find-replace, JSON re-serialization) and
"structural-only" justifications are forbidden. The generated file
`packages/utils/src/lib/i18n.generated.ts` MUST NEVER be edited manually; it is
emitted by `nestjs-i18n` when the API process boots and MUST be regenerated by
running the API after JSON changes, then committed alongside the JSON edits.
Fallback language is `nl_BE`; all three locales MUST stay in sync. The API build
MUST copy the i18n assets into its bundle (asset configuration in the app's
`nest-cli.json`).

**Rationale**: Three-locale parity is a release-blocker; manual edits silently
desync locales and corrupt generated types.

### III. Transactional Mutations

Every GraphQL mutation that performs writes MUST acquire a Sequelize transaction,
commit on success, and roll back on any thrown error. Authorization MUST be
checked inside the resolver via `user.hasAnyPermission([...])` before any write.
Slug-or-UUID lookups MUST branch on `IsUUID(id)` (`findByPk` vs.
`findOne({ where: { slug } })`). Paged results MUST use an inline `@ObjectType()`
returning `{ count: number; rows: T[] }`, and filter operators MUST flow through
`ListArgs` / `queryFixer()` rather than ad-hoc Sequelize `Op` translation.

**Idempotent create mutations**: when a create mutation has a natural uniqueness
key (e.g. `(link, season)` for teams, `(teamId, subEventId)` for enrollments,
`(clubId, playerId, season, type)` for club memberships), it MUST be idempotent
on re-submission. Such mutations MUST return a result `@ObjectType` carrying the
entity's identifiers plus `alreadyExisted: boolean` — `true` when the existing
row matched and no write occurred, `false` when a fresh row was created.
Re-submission MUST NOT throw a "duplicate" error and MUST NOT produce a
duplicate row. Reference implementations: `TeamResult` (`createTeam`) and
`EnrollmentResult` (`createEnrollment`).

**Rationale**: Partial writes corrupt competitive-state data (rankings, draws,
enrollments). Re-submission is common (network retries, browser back-button,
wizard re-saves); silent idempotency prevents spurious user-facing errors and
preserves the client's ability to obtain server-assigned IDs without an extra
lookup. Uniform patterns make resolver behavior predictable and reviewable.

### IV. Resolver Test Discipline

Resolver unit tests MUST follow the reference pattern in
`packages/backend-graphql/src/resolvers/enrollmentSetting/enrollmentSetting.resolver.spec.ts`:
`Test.createTestingModule` with a fake `Sequelize` whose `transaction()` returns
`{ commit, rollback }` stubs; model statics mocked with `jest.spyOn`; no real
database connections; a fake `Player` with a `hasAnyPermission` jest.fn();
`afterEach(jest.restoreAllMocks)` for isolation. CRUD resolvers MUST cover:
query-returns-data, query-returns-empty, mutation-rejects-unauthorized
(`UnauthorizedException`), mutation-handles-not-found (`NotFoundException`),
mutation-success-with-commit, and mutation-rolls-back-on-error. Test files MUST
be co-located beside the source file as `*.spec.ts`.

**Rationale**: Hitting the real database in unit tests is slow and flaky;
without standard cases, regressions in auth and rollback paths slip through.

### V. Frontend Lives in a Separate Repository (NON-NEGOTIABLE)

This repository is backend-only: the NestJS API, Bull workers, and shared
TypeScript packages. The legacy Angular frontend (formerly `apps/badman/` and
`libs/frontend/`) and its end-to-end suites have been REMOVED. Frontend code MUST
NOT be reintroduced into this repository; all frontend work belongs in the
separate frontend repository. The only frontend-adjacent responsibility retained
here is whatever static-asset serving the API itself performs; adding Angular
libraries, an SPA app, or e2e browser suites back into this monorepo MUST be
rejected at review.

**Rationale**: A backend-only repo keeps the Turborepo task graph lean and the
boundary with the active frontend repo unambiguous. Investing in a legacy SPA
inside this repo dilutes effort and reintroduces the drift this removal resolved.

## Technology Stack & Constraints

The following stack is load-bearing and MUST NOT be swapped without a MAJOR
amendment to this constitution:

- **Monorepo**: Turborepo orchestrator over pnpm workspaces. Deployable apps
  live under `apps/`; shared libraries are compiled internal packages under
  `packages/`, imported as `@badman/<name>`. Internal dependencies are declared
  as `workspace:*` and resolved via each package's `exports` map; Nx and
  `tsconfig.paths`-based alias resolution MUST NOT be reintroduced. Task
  orchestration (build/test/lint/dev/deploy) is defined in the root `turbo.json`
  with per-package scripts (no root task logic).
- **Package manager**: pnpm (`pnpm-workspace.yaml`, `pnpm-lock.yaml`,
  `packageManager` pin). npm/`package-lock.json` MUST NOT be reintroduced.
- **Backend runtime**: NestJS on Fastify; API listens on port 5010. Apps build
  with `nest build`.
- **Database**: PostgreSQL via Sequelize; multi-schema (`public`, `event`,
  `ranking`, `security`, `system`, `personal`).
- **Migrations**: `database/migrations/` plain JS, date-prefixed, every `up`/
  `down` wrapped in transactions; `.sequelizerc` reads from `.env`. Migrations
  are independent of the task runner.
- **GraphQL**: Apollo, code-first via `@nestjs/graphql`.
- **Auth**: Auth0 JWKS (RS256) validated by global `PermGuard`; routes are
  public by default; resolvers enforce permissions explicitly. The `@User()`
  decorator MUST be used to access the current user.
- **Queues**: Bull on Redis; queue names are constants in
  `@badman/backend-queue` (`ranking`, `sync`,
  `badminton-belgium-flanders-points`, `badminton-belgium-flanders-places`).
- **Workers**: Lean NestJS apps that import only required modules; processors
  declare `@Processor({ name })` and `@Process(jobName)`.
- **i18n**: `nestjs-i18n` with single `all` namespace, dot-notation keys, three
  locales (`en`, `nl_BE`, `fr_BE`), fallback `nl_BE`.
- **Tests**: Jest, configured per-package; run via `turbo run test`
  (`--filter=<pkg>` for a single package) or `pnpm jest -c <pkg>/jest.config.ts`
  directly. Integration tests are gated behind `RUN_INTEGRATION_TESTS=1`.
- **Versioning/Release**: single repo-wide version derived automatically from
  conventional commits (no hand-authored changelog/version files), producing a
  changelog, git tag, and GitHub release.
- **Formatting**: Prettier; `prettier --check .` MUST pass before merge.

## Development Workflow

- **Branching**: New features branch from `develop` with descriptive names
  (`feat/...`, `fix/...`). Hotfixes MAY branch from the appropriate release
  line. PRs target `develop` unless explicitly stated otherwise.
- **Local services**: `npm run docker:up` starts PostgreSQL, Redis, and pgAdmin.
- **Default dev command**:
  `pnpm dev --filter=api --filter=worker-sync` (Turborepo persistent `dev`
  task).
- **Affected checks**: `turbo run lint test --affected` MUST pass for affected
  packages before review.
- **Migrations**: Authored as date-prefixed JS files under
  `database/migrations/`; applied with `npx sequelize-cli db:migrate`. Both
  `up` and `down` MUST be implemented and transactional.
- **Seeding**: `npm run seed:test-data` for local fixture data.
- **Code review gates**: Reviewers MUST verify (a) constitution compliance,
  (b) translation changes routed through `translation-manager` and
  `i18n.generated.ts` regenerated, (c) mutations transactional and
  authorization-checked, (d) tests follow the resolver pattern, (e) no frontend
  code reintroduced into this repository.

## Governance

This constitution supersedes ad-hoc conventions and informal guidance.
`AGENTS.md` (and its `CLAUDE.md` symlink) is the runtime guidance file for AI
assistants and MUST remain consistent with this document; whichever was edited
last wins, and the other MUST be updated in the same PR.

**Amendment procedure**: Any change is proposed via a PR that edits
`.specify/memory/constitution.md`, updates the Sync Impact Report comment at
the top of the file, and synchronizes affected templates under
`.specify/templates/` and `AGENTS.md`. At least one maintainer review is
required to merge.

**Versioning policy** (semantic):

- **MAJOR**: Backward-incompatible removal or redefinition of a principle, or
  a stack swap listed in Technology Stack & Constraints.
- **MINOR**: A new principle/section added, or material expansion of guidance.
- **PATCH**: Clarifications, wording fixes, typo corrections, non-semantic
  refinements.

**Compliance review**: Every PR description MUST state whether it touches a
principle area (i18n, mutations, models, resolver tests, frontend boundary); if
it does, the author MUST cite the relevant principle. Violations MUST either
be fixed before merge or justified in `Complexity Tracking` of the feature's
implementation plan.
